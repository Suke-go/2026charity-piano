# 要件定義書

## 1. 文書情報
- 文書名: コンサート配信システム設計書
- 版数: v1.0
- 作成日: 2026-04-03
- 対象: インターネット配信基盤、視聴ページ、コメント機能、管理機能

---

## 2. 背景
本システムは、会場で生成される映像をインターネット経由で安定配信しつつ、視聴者からのコメントをリアルタイムに受け付けて表示したり、作品に取り入れるための基盤である。

対象イベントでは以下が重要である。
- 配信映像を安全に外部公開できること
- 視聴者コメントをリアルタイムに受け付けられること
- 不適切コメントや荒らしへの基本的な防御があること
- 実装・運用の複雑さを抑えつつ、本番運用に耐える構成であること

本計画では、低遅延を最優先するのではなく、**約5秒程度の遅延で安定配信すること**を優先する。

---

## 3. 目的
本システムの目的は以下である。

1. 会場の映像・音声をインターネットへ安定配信する
2. 配信ページ上で視聴者コメントをほぼリアルタイムに表示する
3. コメント投稿の停止、削除、制御を運用者が行えるようにする
4. bot や過剰投稿などによる負荷増大を抑制する
5. 配信基盤とコメント基盤を分離し、障害時の影響範囲を限定する
6. 実際のコメントを取得し、コメントを流すインタラクションに反映させる

---

## 4. スコープ

### 4.1 対象範囲
- 会場からのライブ映像配信
- 視聴者向け配信ページ
- コメント投稿・配信・保存機能
- コメント制御用の管理機能
- Cloudflare を利用したエッジ配信・防御・永続化基盤

### 4.2 対象外
- WebRTC を用いた超低遅延配信
- Media over IP
- 録画映像の再配信機能
- 独自動画配信プロトコルの開発
- 高度な機械学習ベースのモデレーションを必須要件とすること
- 会場内LAN/VLAN詳細設計そのもの（別のディレクトリ）

---

## 5. 前提条件・制約
- 映像入力は **ATEM Mini Pro** を使用する
- 想定レイテンシは **約5秒程度** で許容する
- 動画アーカイブは、当面 **カメラ側またはATEM側で保存**する
- 録画映像のオンデマンド再配信は本フェーズでは実装しない
- 配信ページは YouTube Live 風の体験を参考にしつつ、必要機能のみ実装する
- インフラは Cloudflare サービス群を中心に構成する
- セキュリティ要件は、特に以下を重視する
  - 配信ジャックの防止
  - コメントAPIやフロントへの過剰負荷攻撃の抑制

## 6. 利用者・関係者

### 6.1 視聴者
- 配信ページを閲覧する
- コメントを投稿する
- コメントの表示を閲覧する

### 6.2 運用者
- 配信状態を確認する
- コメントの停止・再開を行う
- 不適切コメントを削除・非表示にする
- 告知コメント等を管理する

### 6.3 管理者
- Cloudflare 各種設定を管理する

## 7. システム概要

### 7.1 全体方針
本システムは、以下の3系統に分離して構成する。

1. **動画配信系**
   - 会場映像を Cloudflare Stream に送出し、視聴者へ配信する
2. **コメント系**
   - 視聴者コメントを Workers + Durable Objects で受け、リアルタイム同期する
3. **保存・管理系**
   - コメントデータやログを D1 / R2 に保存し、管理画面から制御する

### 7.2 採用サービス
- Cloudflare Pages
- Cloudflare Workers
- Cloudflare Stream
- Cloudflare Durable Objects
- Cloudflare D1
- Cloudflare R2
- Cloudflare Queues
- Cloudflare WAF
- Cloudflare Turnstile（必要時のみ。視聴者 UX への影響を許容できる場合に限る）
- Cloudflare Access

---

## 8. アーキテクチャ方針

### 8.1 動画配信系
- 会場の映像・音声は ATEM Mini Pro から送出する（正確にはATEM ProをつないだPCより創出する）
- 配信先は Cloudflare Stream Live とする
- 視聴ページには Stream プレイヤーを埋め込む
- 動画配信経路とコメント経路は分離する

### 8.2 コメント系
- 視聴ページからコメントを投稿する
- 投稿先は Workers API とする
- 各イベントのコメントルームは Durable Objects により管理する
- コメント投稿は HTTP POST、視聴者への配信は SSE ベースで行う

### 8.3 保存系
- コメント本文、時刻、状態、削除フラグ等は D1 に保存する
- 監査ログ、JSONエクスポート、付随アセットは R2 に保存する
- 投稿後の非同期処理は Queues を介して行う

### 8.4 管理系
- 管理画面は Cloudflare Access により保護する
- コメント停止、スローモード、削除等の操作を管理画面から行う

## 9. 機能要件

### 9.1 配信ページ表示機能
- 視聴者はブラウザから配信ページにアクセスできること
- ページには以下が表示されること
  - イベントタイトル
  - 動画プレイヤー
  - コメント一覧
  - コメント投稿フォーム
  - コメント停止中等の状態表示

### 9.2 ライブ動画表示機能
- 視聴者は Cloudflare Stream 経由のライブ映像を視聴できること
- 動画は配信ページ内に埋め込まれること
- 動画再生は PC / スマートフォンの一般的ブラウザで利用可能であること

### 9.3 コメント投稿機能
- 視聴者はコメントを投稿できること
- コメント投稿時に bot 対策の検証を行うこと
- コメントは投稿後、可能な限り短時間で他視聴者に反映されること

### 9.4 コメント一覧表示機能
- 新着コメントが配信ページにリアルタイム反映されること
- コメントは時系列で表示されること
- 運用上必要に応じて固定コメントを表示できること

### 9.5 コメント制御機能
- 運用者はコメント投稿状態を以下のモードで切り替えられること
  - OPEN: 通常投稿可能
  - SLOW: 一定時間に1回のみ投稿可能
  - CLOSED: 投稿不可
- 運用者は特定コメントを削除または非表示にできること
- 運用者は不適切コメントを一覧から除外できること

### 9.6 自動モデレーション機能（基本）
- システムはNGワード判定を行えること
- システムは過剰連投を検知し制限できること
- システムはURL投稿など運用上危険な形式を制限できること
- 自動判定されたコメントを保留または非表示候補として扱えること

### 9.7 コメント自動消去機能
- 前面表示用コメントは一定時間経過後に自動的に消去できること
- コメント自動消去は表示上の処理であり、保存データは保持してよい

### 9.8 コメント保存機能
- コメント本文
- 投稿時刻
- イベントID
- 投稿状態
- 削除フラグ
- モデレーション理由

を保存できること。

### 9.9 管理画面機能
- 管理者ログイン後に利用できること
- コメント停止/再開ができること
- コメント削除ができること
- ルーム状態やモード状態を確認できること

## 10. 非機能要件

### 10.1 可用性
- 配信系とコメント系は障害分離されていること
- コメント機能障害時も、動画視聴は継続できる構成とすること

### 10.2 性能
- 動画視聴は約5秒程度の遅延で許容する
- コメント反映は視認上リアルタイムに近い体験を目指す
- コメント受付APIは短時間の集中アクセスに耐えられること

### 10.3 拡張性
- イベントごとにコメントルームを分けられること
- 将来的にリアクション、固定コメント、翻訳等の機能追加が可能であること

### 10.4 保守性
- 動画配信・コメント・保存・管理の責務が分離されていること
- Cloudflare ダッシュボード上で最低限の設定変更が可能であること

### 10.5 運用性
- 運用者が本番中にコメント停止を即時実行できること
- モデレーション作業がブラウザ上で完結すること

---

## 11. セキュリティ要件

### 11.1 配信ジャック防止
- 配信キーは公開ページ・クライアント側に露出しないこと
- 配信設定情報は会場の送出系機器に限定して保持すること
- 管理系情報は一般公開ページから分離すること

### 11.2 負荷攻撃対策
- コメント投稿APIに WAF / Rate Limiting を適用すること
- 投稿フォームは視聴者の即時入力を優先し、Cloudflare Turnstile は既定では表示しないこと
- Turnstile は `PUBLIC_COMMENT_TURNSTILE_REQUIRED=true` の明示設定時のみ使うこと
- Turnstile を有効化する場合は、フロントエンド UI、CSP、UX 影響をあわせて再確認すること
- 投稿はサーバ側で本文長、room state、slow mode、moderation を検証すること

### 11.3 管理画面保護
- 管理画面は Cloudflare Access 配下とすること
- 認証されていない利用者は管理画面へアクセスできないこと

### 11.4 入力値検証
- コメント文字数制限を設けること
- 禁止形式の入力値を拒否できること
- 危険な文字列やURLを制御できること

## 12. データ要件

### 12.1 D1 保存対象
- event_id
- comment_id
- user_session_id
- comment_text
- server_received_at
- display_status
- moderation_status
- deleted_flag
- moderation_reason

### 12.2 R2 保存対象
- コメントエクスポートデータ
- 監査用ログファイル
- 補助アセット

### 12.3 保存方針
- コメント本文は D1 を正本とする
- 大きなファイルやJSONアーカイブは R2 に保存する

### 12.4 コメント timestamp 仕様
- 本フェーズで採用する timestamp は **server_received_at** のみとする
- **server_received_at** は、コメント投稿APIを受けた Worker が付与するサーバ時刻とする
- クライアント端末のローカル時刻は保存しない
- 動画プレイヤーの再生位置に基づく timestamp は本フェーズでは保存しない
- 保存形式は ISO 8601 形式のUTC文字列とする
  - 例: `2026-04-03T15:24:18.231Z`
- コメントの表示順およびログ上の正本時刻は **server_received_at** を基準とする

### 12.5 timestamp 採用理由
- 実装が最も簡単であること
- クライアント時刻のずれや改ざんの影響を受けにくいこと
- 後から時系列検索やログ確認を行いやすいこと
- 将来的に動画時刻や連番を追加しても互換性を保ちやすいこと

### 12.6 API/DB 反映方針
- コメント投稿リクエストでは、クライアントは本文とイベントIDのみ送信すればよい
- Worker は受信時に comment_id と **server_received_at** を生成する
- D1 保存時には少なくとも以下を保持する
  - event_id
  - comment_id
  - user_session_id
  - comment_text
  - server_received_at
  - display_status
  - moderation_status
  - deleted_flag
  - moderation_reason

## 14. 運用要件

### 14.1 本番前
- Stream 側のライブ入力を作成・確認する
- ATEM 側の送出設定を確認する
- コメントAPI、WAF / Rate Limiting、room state、Access の設定確認を行う
- テスト視聴とテスト投稿を実施する

### 14.2 本番中
- 配信映像が正常に表示されていることを確認する
- コメント投稿状況を監視する
- 必要に応じてコメント停止・削除を実施する

### 14.3 本番後
- 必要なコメントログを保全する
- 管理モードを閉じる
- 不要な一時設定を解除する

## 15. 受け入れ基準
- 視聴ページでライブ映像が表示されること
- 視聴者がコメントを投稿できること
- コメントが他視聴者へ反映されること
- 管理画面からコメント停止ができること
- 管理画面からコメント削除ができること
- rate limiting とサーバ側投稿制御が有効であること
- Turnstile は必要時のみ有効化され、通常の視聴者投稿 UX を妨げないこと
- 管理画面が Access により保護されていること
- コメントデータが D1 に保存されること

## 18. 実装方針まとめ
本フェーズでは、配信基盤を過度に複雑化せず、以下の方針で実装する。

- 動画は Cloudflare Stream に任せる
- フロントは Pages に配置する
- コメントは Workers + Durable Objects で扱う
- 保存は D1 / R2 に分ける
- 防御は WAF / Rate Limiting / サーバ側検証 / Access で行い、Turnstile は必要時のみ追加する
- WebRTC や録画再配信は採用しない

この方針により、実装量と運用負荷を抑えつつ、本番イベントで必要な機能と安全性を両立する。
